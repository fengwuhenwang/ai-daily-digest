# 版本回滚流程指南

## 📋 触发条件

执行回滚前需满足以下任一条件：
- ⚠️ 系统出现严重bug或崩溃
- ⚠️ 用户对新版本不满意
- ⚠️ 回归测试失败
- ⚠️ 关键功能异常
- ⚠️ 性能严重下降

---

## 🔄 标准回滚步骤

### 步骤1：评估回滚必要性
```
问题描述：[具体问题]
影响范围：[评估影响]
回滚必要性：是/否
目标版本：V{X.Y.Z}
```

### 步骤2：准备回滚
```bash
# 确认目标版本快照存在
ls -la versions/v{X.Y.Z}_snapshot.md

# 备份当前版本（防止回滚失败）
cp core/jarvis_v{current}.txt versions/v{current}_pre_rollback.md

# 记录回滚计划到 CHANGELOG.md
```

### 步骤3：执行回滚
```bash
# 读取目标版本快照
cat versions/v{X.Y.Z}_snapshot.md > core/jarvis_v{current}.txt

# 更新 SOUL.md 中的系统版本号
# 找到 "系统升级 V{current}" 行，更新为新版本信息
```

### 步骤4：验证回滚
- ✅ 基础功能测试通过
- ✅ 关键场景测试通过
- ✅ 性能表现正常
- ✅ 无新增异常

### 步骤5：记录回滚
在 CHANGELOG.md 中添加：
```markdown
### 回滚记录
- 回滚至：V{X.Y.Z}
- 回滚时间：YYYY-MM-DD
- 回滚原因：[原因描述]
- 回滚版本号：V{current} → V{X.Y.Z}
```

---

## 🚨 紧急回滚（Hotfix）

当系统完全无法运行时：

1. **立即回退到上一稳定版本**
   ```bash
   # 直接使用 git 历史回退
   git checkout HEAD~1 core/
   ```

2. **事后补充文档**
   - 在 CHANGELOG.md 记录回滚原因
   - 创建 hotfix 版本修复问题
   - 重新测试并发布

## ⚠️ 注意事项

### 回滚前检查清单
- [ ] 确认目标版本快照完整
- [ ] 备份当前版本
- [ ] 记录回滚原因
- [ ] 准备好回滚失败的后备方案

### 回滚后验证清单
- [ ] 核心功能正常
- [ ] 模块加载正确
- [ ] 无逻辑冲突
- [ ] 性能可接受

### 禁止事项
- ❌ 未备份直接回滚
- ❌ 跳过验证步骤
- ❌ 频繁回滚（表明版本管理有问题）
- ❌ 在生产环境直接修改核心文件

---

## 📊 回滚决策树

```
发现问题
    ↓
严重程度评估
    ↓
    ├─ 低严重性 → 记录到 CHANGELOG → 下版本修复
    ├─ 中严重性 → 快速补丁（patch版本）
    └─ 高严重性 → 回滚到上一稳定版本
```
